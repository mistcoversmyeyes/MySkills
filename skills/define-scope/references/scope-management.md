# Reference · 范围管理（Scope Management）速查

范围管理的核心不是"写文档"，而是**保证所有相关方对"做什么 / 不做什么"有一致且可追溯的认知**。文档只是载体。

## 范围管理五大过程（PMBOK 视角）

| 过程 | 做什么 | 本 skill 中对应的动作 |
|------|--------|----------------------|
| **1. 规划范围管理** | 定义怎么管理范围（流程、模板、审批机制） | 使用本 skill 本身就是这一步 |
| **2. 收集需求** | 访谈、问卷、头脑风暴，得到原始需求 | SKILL.md 阶段 1（目标澄清） |
| **3. 定义范围** | 把需求翻译成项目范围说明书（in/out-of-scope、假设、约束） | SKILL.md 阶段 2（约束与假设）+ scope.md 第 4 节 |
| **4. 创建 WBS** | 把范围分解为可管理的工作包 | SKILL.md 阶段 4（WBS） |
| **5. 确认 + 控制范围** | 验收交付物；识别并处理范围变更 | SKILL.md 第 5 节（后续跟踪） |

## 三类常见陷阱

### 陷阱 1：范围蔓延（Scope Creep）

没有明确批准就悄悄加入新需求。应对：所有新需求走"变更记录"表，显式做"接受 / 拒绝 / 延后"决策。

### 陷阱 2：镀金（Gold Plating）

执行者自作主张做了超出范围的"额外好事"，挤占了主线资源。应对：任何觉得"我顺手做了比较好"的冲动，先写进 backlog，等下周期对齐后再说。

### 陷阱 3：范围不足（Scope Under-specification）

范围写得太笼统（"优化性能"），导致验收时各说各话。应对：每个交付物必须有可度量的验收标准——"性能提升" ≠ 验收标准，"vm-exit/sec 在基准场景下下降 20%" 才是。

## 需求追溯矩阵（RTM，可选）

当交付物较多、来源较杂时，建一份追溯矩阵，把每条需求追溯到：源头（谁提的）→ 对应的范围条目 → 对应的 WBS 工作包 → 对应的验收测试。这样后期"为什么做这个"永远查得到。

## 三个你必须对用户逼问的问题

1. "这件事**不做什么**？" — 逼出 out-of-scope。
2. "**怎样算做完了？**" — 逼出验收标准。
3. "如果**只能做一半**，优先做什么？" — 逼出真正的 Must。
