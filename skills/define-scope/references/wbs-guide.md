# Reference · WBS（Work Breakdown Structure）指南

WBS 是把项目**最终可交付物**逐层分解为更小、更可管理的工作单元的层次结构。它回答的问题是：**"为了交付这个东西，需要做哪些具体的活？"**

## 核心原则

### 1. 100% 法则（最重要）

> 子节点应**完整地且仅仅**覆盖父节点的工作——不多不少。

- "不少"：所有必要工作都必须在 WBS 中出现，漏一项就漏一项工时。
- "不多"：不应出现不属于父节点的工作。

实用检查：合上 WBS，问"把所有叶子节点都完成之后，父节点的交付物是不是就完成了？"。若否，说明不完整；若叶子节点有明显属于别的方向的，说明越界。

### 2. 按交付物分解（Deliverable-Oriented），不是按动作

**好的分解**：按"要产出的东西"组织。
```
1. 性能测试报告
   1.1 基准环境搭建
   1.2 基准数据采集
   1.3 报告撰写
```

**不好的分解**（按活动组织，容易漏交付物）：
```
1. 开会
2. 写代码
3. 测试
```

### 3. MECE（相互独立、完全穷尽）

- Mutually Exclusive：兄弟节点之间不重叠。
- Collectively Exhaustive：兄弟节点合起来覆盖父节点全部。

如果 1.1 和 1.2 有交集，迟早两个活都有人在做、浪费；如果合起来不等于 1，就有活没人认领。

### 4. 8/80 规则（粒度参考）

叶子工作包的工作量介于 **8 小时 到 80 小时**（即 1–10 个工作日）之间。

- 小于 8 小时：管理成本比执行成本还高，合并到上一级。
- 大于 80 小时：不可控，再往下分一层。

**本 skill 的务实调整**：个人开发者场景下建议 **4 小时 – 2 人天**，更贴合日常节奏。

### 5. 叶子节点可估时、可验收

每个叶子节点都应该回答：
- 工作量多少？
- 谁做？
- 做完了怎么验证？

## 常见分解模式（可组合使用）

| 模式 | 适用场景 | 示例 |
|------|----------|------|
| **按交付物** | 有清晰交付物清单的项目 | 报告、模块、PR 各为一级 |
| **按阶段** | 调研类、有明显生命周期 | 调研 → 原型 → 评估 → 落地 |
| **按子系统/模块** | 大型代码改造 | VMM / HostDriver / Frontend 各一级 |
| **按角色/组件** | 前后端或硬件软件协同 | 前端 / 后端 / 数据 |

一般**顶层按"交付物"或"阶段"**，下一层再视情况按模块展开，避免一上来就按团队组织结构分解。

## 不确定节点的处理：Spike

当某个工作包因为信息不足而无法估时时（典型情况：调研、选型、新技术验证），**不要硬估**。做法：

- 把该节点标记为 `[SPIKE]`。
- 给它一个**时间盒**（timebox），比如 "1 天 / 3 天"。
- 明确这个 spike 的**决策产出**——做完之后要回答什么问题、做什么决策。
- spike 结束后再回头对原节点做正式分解。

例：`1.2 [SPIKE, 1天] 调研 VMware 半虚拟化接口方案 → 决策产出：是否值得在 VBox 中参考其方案`

## WBS 字典（可选）

如果工作包名称本身不够说明问题，可以维护一份"WBS 字典"，对每个工作包补充描述：范围、验收标准、负责人、依赖、资源。本 skill 的 `wbs-template.md` 已经通过表格字段把这部分关键信息内嵌了，无需单独建字典文件，除非项目规模很大。
