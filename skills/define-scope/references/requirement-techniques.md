# Reference · 需求与目标表达技巧

这份速查列出在 SKILL.md 阶段 1–3 中常用的表达工具。它们的共同目的是——**让模糊的方向变成可验证的陈述**。

## 1. SMART 目标

目标的五个维度：

| 字母 | 含义 | 反例 → 正例 |
|------|------|------------|
| **S**pecific 具体 | 说清楚"做什么" | "提升性能" → "降低 VMX 场景下 vm-exit/sec" |
| **M**easurable 可度量 | 有量化指标 | "变快" → "在 X 基准下吞吐 +20%" |
| **A**chievable 可达成 | 在资源约束内可行 | "三天内写完一个操作系统" ✗ |
| **R**elevant 相关 | 与更大目标对齐 | 检查："这条目标有助于业务 / 技术北极星吗？" |
| **T**ime-bound 有时限 | 有完成期限 | "尽快" → "Q2 内" |

## 2. MoSCoW 优先级

| 等级 | 含义 | 判断问题 |
|------|------|----------|
| **M**ust | 本周期必须做，不做则失败 | "没这个，整件事还成立吗？" |
| **S**hould | 应该做，但即便没做也不算失败 | "没这个会很可惜，但不会翻船" |
| **C**ould | 有余力才做 | "锦上添花" |
| **W**on't | 本次明确不做（**记录下来**） | "这次不做，下次再议" |

用它的诀窍：**逼用户把超过 30% 的条目从 Must 降级**。否则"一切都是 Must"等于没有优先级。

## 3. Kano 模型（可选）

从用户视角衡量功能价值：

- **基本型**（Must-be）：没有会不满，有了也不会特别开心（比如浏览器能打开网页）。
- **期望型**（One-dimensional）：越多越满意（比如加载速度）。
- **兴奋型**（Attractive）：没有也无妨，有了会惊喜（比如某个智能补全）。
- **无差异型**（Indifferent）：用户根本不在意。
- **反向型**（Reverse）：做了反而招嫌。

对开发者场景的简化用法：区分"必须交付"（基本+期望）和"加分项"（兴奋）。无差异和反向的果断砍掉。

## 4. INVEST（用户故事 / 工作包的质量检查）

一个好的工作包应满足：

- **I**ndependent 独立（与其它工作包尽可能解耦）
- **N**egotiable 可协商（描述是方向，具体做法可讨论）
- **V**aluable 有价值（能独立交付价值，而不是"半截子"中间产物）
- **E**stimable 可估时
- **S**mall 足够小（符合 8/80 或 4 小时–2 天规则）
- **T**estable 可测试/可验收

当你在 WBS 里写下一个工作包，挨个过一遍这 6 个字母。最常违反的是 V（有价值）——很多"内部重构一半"的工作包做完了也说不上交付了什么。

## 5. 验收标准的三种常用写法

### Given/When/Then

```
Given <前置条件>
When <触发操作>
Then <可观察结果>
```

例：*Given 虚拟机已启动并运行 UnixBench 基准测试；When 持续运行 10 分钟；Then 每秒 vm-exit 数较基线下降 ≥ 20%。*

### 可量化指标式

直接列"哪些数字达到多少"。适合性能/可观测性类交付物。

### 清单勾选式

适合文档类、功能开关类——列一组 checklist，全勾上即完成。
