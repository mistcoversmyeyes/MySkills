# Reference · 进度管理基础

本 skill 的进度部分只做**骨架级**支持——识别里程碑、依赖、关键路径，不做精细排期和资源平衡（那是专门的项目管理工具的事）。

## 1. 里程碑（Milestone）

里程碑是**零工时的检查点**——标志某个阶段性成果到位。例如："基准环境搭建完成"、"调研报告评审通过"、"性能优化代码合入主干"。

**好里程碑的特征**：
- **二元的**（要么达到、要么没达到，没有"80% 达到"）。
- **有可见的证据**（文档、测试报告、PR 链接）。
- 数量适中（通常一个工作方向 3–5 个，最多不超过 7 个，否则失去标志意义）。

## 2. 依赖类型（Predecessor Relationships）

四种依赖关系：

| 缩写 | 名称 | 含义 | 典型场景 |
|------|------|------|----------|
| **FS** | Finish-to-Start | A 做完才能开始 B（最常见） | "环境搭建完才能跑测试" |
| **SS** | Start-to-Start | A 开始了 B 才能开始 | "正式编码开始后，文档撰写才能同步启动" |
| **FF** | Finish-to-Finish | A 做完 B 才能做完 | "所有开发完成后，最终集成才能完成" |
| **SF** | Start-to-Finish | A 开始 B 才能结束（罕见） | 交班场景 |

默认假设是 FS。只有当两个工作包确实不能 FS 的时候才考虑其它。

另外区分：
- **强制依赖**（Mandatory）：不可违反的客观依赖（代码没写就没法测）。
- **自由依赖**（Discretionary）：团队自己决定的顺序偏好（先做 UI 还是先做 API）。
- **外部依赖**：依赖本项目外的方（上游团队、第三方库发布）。

## 3. 关键路径法（CPM）简介

**关键路径** = 从起点到终点所有路径中总工时最长的那条。

**关键点**：
- 关键路径上的任一工作包延期 → 整体延期。
- **不在关键路径上的工作包有"浮动时间"**（Float / Slack），意思是晚一点做也不影响整体。
- 优化整体工期的第一步永远是：**优化关键路径上的工作包**。对非关键路径上的工作包做优化，对整体无帮助。

**实用做法**（不需要软件辅助）：
1. 把 WBS 叶子画成有向图，节点是工作包，边是 FS 依赖。
2. 从左到右算每个节点的"最早开始 / 最早结束"时间。
3. 从右到左算"最晚开始 / 最晚结束"时间。
4. 浮动时间 = 最晚开始 − 最早开始。浮动时间 = 0 的节点就在关键路径上。

对本 skill 的使用场景（一般一个方向 10–30 个工作包），肉眼看图 + 简单表格就够，不用上专业工具。

## 4. 时间估算的三种常用做法

| 方法 | 怎么算 | 适用 |
|------|--------|------|
| **类比估算** | "上次做类似的花了 3 天，这次应该差不多" | 有历史参考时，快但粗 |
| **三点估算**（PERT） | `(乐观 + 4×最可能 + 悲观) / 6` | 不确定性较高，想要期望值 |
| **Planning Poker / 故事点** | 团队估相对大小，不直接估时间 | 团队协作场景 |

对于个人开发者场景，推荐**类比估算为主、三点估算兜底**——尤其给 spike 节点估时间盒时，用悲观值作为硬性上限。

## 5. 与 TODO 的关系

进度骨架（里程碑 + 关键路径）放在 `wbs.md` 里；日常跟踪用 `todo.md`。**不要把 todo.md 升级成甘特图**——那是用错了工具。
