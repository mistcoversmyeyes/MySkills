# Define Scope — 软件工作方向的范围澄清与任务拆解助手

## 1. 这个 skill 在解决什么问题

用户日常会接到各种"方向级"的活：一个技术调研题、一块代码的重构要求、一个性能优化方向、一个模糊的增量需求。这类任务的共同特点是——**目标不够具体、路径不够确定、边界容易飘移**。如果直接开工，往往做到一半发现"这块是不是我该做的""好像还漏了一个场景""老板说的是不是这个意思"。

范围管理（Scope Management）和工作分解（WBS）这些项目管理手段，本质上就是把"方向"变成"可以被验收的一组交付物"。本 skill 的工作，就是用对话把用户脑子里的方向"榨"成一份可查、可跟踪、可 review 的范围文档。

## 2. 适用边界（必须在第一轮交互后评估）

本 skill 的定位是处理**模糊方向**。如果用户拿来的工作其实很清晰，强行走完整流程反而是过度流程——这会消耗用户耐心且没什么产出价值。

收到用户方向描述后，**第一件事不是开工，而是做一次"复杂度画像"**。向用户明确提出以下 5 个 yes/no 问题（可以一次性全问，也可以合并成一段对话，不要逐条折磨用户）：

1. 这个方向的**目标**能否用一句话清晰描述？（例如"减少 VBox 在 Windows Host 下的 VM Exit 次数"是清晰的；"提升虚拟化性能"是不清晰的）
2. **实现路径**是否已经明确？（比如已经知道改哪几个文件、用什么方案；反之如果还要先调研选型，就不算明确）
3. 预估**总工作量 ≤ 1 人天**？
4. 是否**仅涉及单一模块**、单人即可完成、无跨团队依赖？
5. **验收标准**是否一目了然（例如"某测试用例通过"、"某个按钮能点击"）？

根据"是"的个数分档响应：

### 档位 A：5 题全是"是" —— 劝退 + 二次确认

明确告诉用户：

> 这个任务方向已经足够清晰：目标明确、路径确定、工作量小、验收直观。引入正式范围管理与 WBS 反而是过度流程，会拖慢你而不是帮助你。
>
> **建议直接使用 Plan Mode 把它拆成几条 TODO 执行即可，不必留范围文档。**
>
> 如果你出于归档、交接、对齐等原因仍希望产出一份精简 scope，请回复"继续"。

**必须等用户明确回复"继续"或同类意图后**才进入产出流程（此时走档位 B 的精简版）。不要擅自开工。

### 档位 B：3–4 题是"是" —— 降级产出（scope + todo）

说明一句"这个方向基本清晰，但[点出还有哪几项不确定]，我给你出一份精简的 scope + todo，不再做详细 WBS 分解"，然后只生成：
- `scope.md`（精简版，控制在 1 页内）
- `todo.md`

跳过 WBS。

### 档位 C：≤ 2 题是"是" —— 完整三件套

走下面第 3 节的完整交互流程，最终产出：
- `scope.md`（范围说明书）
- `wbs.md`（工作分解结构）
- `todo.md`（下一阶段可执行 TODO 清单）

## 3. 完整交互流程（档位 C）

采用**四阶段分步激发**，每一阶段只聚焦一件事，避免一次性甩给用户一大堆问题。每阶段结束后都**先把你理解到的内容复述给用户确认**，再进入下一阶段。

### 阶段 1 — 目标澄清（WHY / WHAT）

提问要点（挑对方向的问，不必全问）：
- 这个工作的**业务/技术动机**是什么？为什么现在做？
- 期望的**最终成果**是什么形态？（调研报告？代码 PR？性能数字？原型 Demo？）
- 成果的**使用者 / 受众**是谁？（团队内部、上级、外部客户）
- 是否有明确的**时间窗口**或里程碑？

关键点：把用户说的模糊词翻译成 SMART 形式。查阅 `references/requirement-techniques.md` 了解 SMART 与 INVEST 的用法。

### 阶段 2 — 约束与假设（BOUNDARY）

提问要点：
- 技术栈 / 平台 / 环境约束（操作系统、语言、版本、依赖）
- 显式**排除项**（"不需要做 XX"），这比 in-scope 更重要
- 资源约束（人力、硬件、时间、外部依赖）
- 前置**假设**（"假定 Hyper-V 不开启"、"假定 Guest 是 Linux 4.x"）
- 依赖的上游方（是否要等别的团队、别的 PR、别的数据）

关键点：排除项（out-of-scope）和假设（assumption）都必须写入 scope.md，否则后期范围蔓延时无据可依。

### 阶段 3 — 交付物与验收（DELIVERABLE）

提问要点：
- 列出每一个**具体交付物**（文档、代码模块、实验数据、报告、PR）
- 每个交付物的**验收标准**是什么？（谁签字、基于什么测？）
- 是否分阶段交付？（中期 / 最终）

关键点：套用 MoSCoW（Must / Should / Could / Won't）给交付物分优先级，见 `references/requirement-techniques.md`。

### 阶段 4 — 分解与排期（WBS & SCHEDULE）

这一阶段开始动笔。基于前三阶段结果，按以下步骤把方向分解成工作包：

1. **按交付物驱动分解**（不是按组织结构），遵循 100% 法则——子任务完整并且仅仅覆盖父任务，不多不少。见 `references/wbs-guide.md`。
2. 控制粒度：叶子工作包的工作量建议在 **4 小时 – 2 人天**（8/80 规则的务实版本）。粒度过粗则不可跟踪，过细则成本大于收益。
3. 识别**关键依赖**与**里程碑**。进度相关概念见 `references/schedule-basics.md`。
4. 识别**高不确定性节点**（标注 "spike / 调研 / 需要验证"），这些节点不要硬排工时，标记"先做 1 天时间盒调研"。

完成后，把 WBS 中**当前可立刻开工**的叶子节点抽出来，生成 `todo.md`。`todo.md` 只放"下一轮要做的"，不是把全部 WBS 搬一遍——WBS 负责全景，TODO 负责当期执行。

## 4. 产出位置与文件

产出默认放在**用户当前工作目录**下的 `docs/scope/<topic>/`，一个工作方向一个子目录。`<topic>` 从用户方向描述中提炼一个 kebab-case 短标识，先跟用户确认后再建目录（例如 `vbox-cpu-paravirt-research`、`mumu-storage-refactor`）。

一个典型目录是这样：

```
docs/scope/<topic>/
├── scope.md   # 范围说明书
├── wbs.md     # 工作分解结构（档位 C 才有）
└── todo.md    # 下一阶段可执行 TODO
```

三份文件的模板分别在：
- `assets/scope-template.md`
- `assets/wbs-template.md`
- `assets/todo-template.md`

**写文件时从模板拷贝起手，然后填充**，不要从零生成——模板里的字段本身就是提醒你不要漏项的清单。

## 5. 后续跟踪（范围锚定）

本 skill 不仅"开局拆解"，也承担后续的**范围锚定**职责。当用户在后续对话中说"老板又加了一个需求"、"我想把 XX 也做了"、"这个做完发现还要改 YY"时，应当：

1. 打开对应 `scope.md`，定位到"范围边界"与"排除项"章节。
2. 判断新需求属于：
   - **in-scope 的澄清**（原本就在范围内，只是描述不够细）→ 直接在 scope.md 补充描述。
   - **合理的范围扩展**（值得做，且不破坏主线）→ 走一次"范围变更"：在 scope.md 末尾的"变更记录"表中登记（日期 / 变更内容 / 影响 / 决策），同步更新 WBS 与 TODO。
   - **范围蔓延**（scope creep，偏离主线、挤占资源）→ **提醒用户**：这条需求不在原范围内，建议要么明确接受并调整排期，要么放进 `docs/scope/<topic>/backlog.md`（如无则新建）延后处理。

这一步的核心价值是——**帮用户把"我可能要做 XX"变成一次有意识的决定**，而不是被动被需求推着走。

## 6. 写作风格建议

- **所有产出文档用中文**，与仓库其他 `docs/` 保持一致。
- scope.md 第一段必须是"一句话目标"——如果你写不出一句话，说明阶段 1 没做透，回去重做。
- 宁可承认"此处尚不确定，需要 1 天 spike"，也不要在 WBS 里编一个看起来很整齐但其实没依据的分解。
- 用户说话含糊时，**不要猜**，回到阶段 1 的提问——把模糊处重新澄清一次是该 skill 的主要价值。
