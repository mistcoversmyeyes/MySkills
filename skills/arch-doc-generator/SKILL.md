# 架构文档生成器

> **目标子系统**: $subsystem
> **输出路径**: $output_path（如未指定，由用户确认）

## 核心框架

本 Skill 采用 **4+1 架构视图模型**，将文档组织为三个结构维度：

| 结构维度 | 视图 | 关注点 | 典型 UML 图 |
|----------|------|--------|-------------|
| **静态结构** | Logical View | 核心抽象、类/结构体关系 | 类图、对象图 |
| **静态结构** | Development View | 源文件组织、模块依赖、构建单元 | 包图、组件图 |
| **动态结构** | Process View | 运行时调用流程、并发交互、状态变迁 | 时序图、活动图、状态图 |
| **分配结构** | Allocation View | 软件到硬件/进程/节点的映射 | 部署图、组件部署图 |

并非每个子系统都需要全部视图。根据子系统特点选择适用的视图——这一点很重要，因为强行为一个简单模块画部署图只会产生噪音而非信息。

## 工作流程

### Phase 1: 理解目标子系统

在动手画图之前，先搞清楚要文档化的对象：

1. **阅读源码**：至少读取核心头文件和主要实现文件，理解关键数据结构、公开 API、内部实现细节
2. **识别边界**：确定子系统的公开接口和内部实现的分界线
3. **梳理依赖**：找出与其他模块的依赖关系（include、调用、回调等）
4. **确定范围**：与用户确认需要覆盖的深度（仅公开 API？还是包含内部实现细节？）

### Phase 2: 选择适用视图

根据子系统特点决定哪些视图有价值：

- **所有子系统**都应有 Logical View（类图是最基本的结构化工具）
- **有多文件/多模块**的子系统需要 Development View
- **有运行时交互**的子系统需要 Process View（时序图或活动图）
- **跨进程/跨节点/跨特权级**的子系统需要 Allocation View

主动建议哪些视图适合，让用户确认或调整。

### Phase 3: 创建文档

#### 文档骨架

使用 `${CLAUDE_SKILL_DIR}/assets/template.md` 作为起始模板。该模板包含完整的三级骨架结构，其中 `$subsystem` 变量会自动替换为用户传入的子系统名称。模板中的 `<!-- TODO -->` 注释标记了所有需要填充的位置。

复制模板到 `$output_path`，然后逐步填充各 TODO 位置的内容。

#### 章节标题命名规范

每个小节标题格式统一为：`## N.M 视图名 — 描述性标题`

**示例：**
- `## 1.1 Logical View — 核心类图`
- `## 1.2 Logical View — 宏层级结构图`
- `## 1.3 Development View — 模块组织与源文件依赖`
- `## 2.1 Process View — 日志调用时序`
- `## 3.1 Allocation View — Ring-0 / Ring-3 组件部署`

### Phase 4: 绘制 PlantUML 图

所有 UML 图使用 PlantUML 语法，嵌入在 Markdown 代码块中。

#### PlantUML 编写规则

这些规则是从大量 PlantUML 渲染失败中总结出来的，遵守它们可以避免浪费时间在语法调试上：

1. **组件命名用双引号**，不要用方括号 `[...]`
   ```plantuml
   ' ✅ 正确
   component "My Component" as MyComp
   ' ❌ 错误 — 方括号内含 \n 或特殊字符容易导致解析错误
   component [My Component\nSub info] as MyComp
   ```

2. **不要在 component 内嵌 portin/portout**
   ```plantuml
   ' ✅ 正确 — 把端口信息写进组件描述
   component "Log Engine\n---\ng_pRelLogger\ng_pLogger" as Engine

   ' ❌ 错误 — 很多 PlantUML 版本不支持
   component "Log Engine" as Engine {
       portin "g_pRelLogger" as pRel
   }
   ```

3. **`skinparam componentStyle` 使用 `rectangle`**，不要用 `uml2`（兼容性更好）

4. **每张图起名**：`@startuml DiagramName` 让图表有辨识度

5. **中文注释用 `note` 语法**，不要直接写在元素名称中（超长中文字符串容易溢出）

6. **枚举/类的 `...` 省略号**：当成员过多时，用 `...` 表示省略即可

7. **避免过于复杂的单张图**：如果一张类图超过 15 个类，考虑拆分成多张图，分别聚焦不同方面

#### 各视图的图表选择指南

**Logical View（静态结构的逻辑面）：**

| 场景 | 推荐图表 | 说明 |
|------|----------|------|
| 核心数据结构及关系 | 类图 | 展示字段、继承、组合、依赖 |
| 宏/API 的分层展开 | 组件图（分层） | 用 package 表示每层，箭头表示展开关系 |
| 接口与实现的关系 | 类图 + interface | 突出 vtable/回调模式 |
| 枚举/标志位体系 | 类图中的 enum | 列出关键值，过多时省略 |

**Development View（静态结构的开发面）：**

| 场景 | 推荐图表 | 说明 |
|------|----------|------|
| 源文件/头文件依赖 | 包图 | 每个文件或目录是一个 package |
| 库/DLL 的组成 | 组件图 | 每个构建产物是一个 component |
| 全局变量/单例 | 包图中标注 | 在实现文件的 package 中列出 |

**Process View（动态结构）：**

| 场景 | 推荐图表 | 说明 |
|------|----------|------|
| 函数调用链 | 时序图 | 最常用，展示完整的请求处理路径 |
| 初始化/启动流程 | 时序图 | 展示创建和注册顺序 |
| 条件分支逻辑 | 活动图 | 适合编译期或运行时的分支决策 |
| 生命周期/模式切换 | 状态图 | 适合有明确状态机的模块 |

**Allocation View（分配结构）：**

| 场景 | 推荐图表 | 说明 |
|------|----------|------|
| 跨进程/跨节点部署 | 部署图 (node) | 每个进程/节点是一个 node |
| 跨特权级 (Ring-0/3) | 部署图 (node) | 每个特权级是一个 node |
| 组件到物理机的映射 | 部署图 | node 内放 component |

### Phase 5: 质量检查

完成文档后，做一轮自检：

1. **结构一致性**：所有章节标题是否遵循 `## N.M 视图名 — 描述` 格式
2. **PlantUML 语法**：是否使用了双引号而非方括号，是否避免了 portin/portout 嵌套
3. **图文对应**：每张图前是否有一句话说明这张图展示的内容
4. **适度详细**：类图中的字段是否足够理解结构但不至于列出每一个字段（对于超大结构体，按功能分组并标注分隔线）
5. **跨引用**：附录中是否列出了关键源文件索引

## 参考示例

`references/example-vbox-logsystem.md` 是一份完整的实际产出示例——VirtualBox 日志系统的架构文档。在生成文档前阅读它，重点关注：

- **文档骨架**：开头的三维度总览表、三个一级标题的组织方式
- **章节标题格式**：`## N.M 视图名 — 描述` 的统一命名
- **类图写法**：如何用分隔线 `--` 对大结构体按功能分组、enum 的省略写法、interface/callback 的表示
- **时序图写法**：`alt`/`else` 分支、`loop` 循环、`activate`/`deactivate` 的使用
- **活动图写法**：编译期条件分支的表达方式
- **部署图写法**：`node` 内嵌 `component`、跨节点虚线依赖

将此示例作为**风格和质量的基准**，但不要机械照搬——每个子系统的图表数量和类型应根据实际情况调整。

## 注意事项

- **先读代码再画图**：不要凭想象编造数据结构或 API，所有图表内容必须来自实际代码
- **图是手段不是目的**：如果某个视图对这个子系统没有增量信息价值，跳过它并说明原因
- **文档语言跟随用户**：用户用中文交流就写中文文档，用英文就写英文文档
- **PlantUML 图内注释**可以用中文或英文，与文档语言一致
- **附录放辅助信息**：关键源文件索引、项目特有的定制说明等
