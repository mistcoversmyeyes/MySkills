# LeetCode 题解生成助手

用于配合 VSCode LeetCode 插件，在 `~/.leetcode/docs/` 目录下创建题解文档。

## 核心工作流程

### 第一步：识别题目

从以下来源确定题号和题目标题：
1. **当前打开的题目文件**（如 `49.字母异位词分组.cpp`）
2. 从文件头部注释解析题号：`@lc app=leetcode.cn id=49`
3. 从文件名提取题号：正则匹配 `(\d+)\.`

### 第二步：获取题目信息

使用 `scripts/fetch_problem.py` 获取题目数据：

```bash
# 从文件解析
python scripts/fetch_problem.py --file /path/to/problem.cpp

# 直接指定题号
python scripts/fetch_problem.py 49
```

脚本支持多种数据源（按优先级）：
1. leetcode-cli（如果已安装）
2. 降级模式：输出基本信息模板，手动补充

### 第三步：创建题解目录和文件

在 `~/.leetcode/docs/` 下创建目录结构：
```
{题号}. {题目标题}/
└── 题解.md
```

例如：
```
49. 字母异位词分组/
└── 题解.md
```

### 第四步：基于模板填充结构

**重要**：使用 `assets/solution-template.md` 作为模板，**严格按照模板的 markdown 结构**生成题解。

模板结构（必须遵守）：
```markdown
# 思路X：{思路名称}

## 解题思路
**核心思想**：{一句话概括}

**算法流程**：
1. {步骤一}
2. {步骤二}
3. {步骤三}

## 复杂度
- 时间复杂度：{分析}
- 空间复杂度：{分析}

## Code
```cpp
{代码实现}
```

**问题/优化方向**：{如果有}

---

# 总结对比
| 思路 | 时间复杂度 | 空间复杂度 | 特点 |
|------|-----------|-----------|------|
...
```

**格式要求**：
- 思路标题：`#`（一级标题）
- 子节标题：`##`（二级标题）
- 必须包含：`## 解题思路`、`## 复杂度`、`## Code` 三个二级标题
- **不要**使用其他格式（如 `**核心思想**` 作为独立段落）

### 第五步：从对话历史提取真实思考过程

**最关键**：不要只写成功的解法，必须记录**完整的思考历程**，包括失败的尝试。

#### 5.1 提取思路演进顺序

从对话历史中按时间顺序提取所有思路，包括：

1. **最自然的想法**（往往是暴力法）
   - 用户的原话："最自然的想法是..."
   - 标记为"思路一"

2. **失败的优化尝试**（如果有）
   - 用户尝试过但失败的方法
   - 记录为什么失败
   - 标记为"思路X（❌ 失败尝试）"
   - 示例：
     ```
     **尝试过程**：
     1. 想到用哈希表优化暴力法中的重复计算问题
     2. 尝试了**字符求和法**：
        ```cpp
        int hash(string str) {
            int hash_val = 0;
            for (char ch : str) {
                hash_val += (int)ch;
            }
            return hash_val;
        }
        ```
     3. 发现问题：会有**哈希碰撞**
        - "ac" = 97 + 99 = 196
        - "bb" = 98 + 98 = 196
        - 不是字母异位词的词被分到了一组！

     **问题/优化方向**：
     - 找不到一个完美的哈希函数...
     - **卡了 6-7 分钟**，一度认为面试要完蛋了
     - **关键突破**：在面试官的引导下发现...
     ```

3. **成功的解法**（最终通过的方法）
   - 记录关键的"恍然大悟"时刻
   - 标记为"思路X"或"思路X（✅ 推荐解法）"

4. **进一步优化**（如果有讨论）

#### 5.2 添加情景感和细节

让题解有**情景感**，记录真实的思考过程：

✅ **好的示例**（有情景感）：
```markdown
这是我最自然的想法：对于每个词，找到它的字母异位词，然后放入结果临时数组中，然后删除选中的词。

**尝试过程**：
1. 想到用哈希表优化暴力法中的重复计算问题
2. 尝试了**字符求和法**：
3. 发现问题：会有**哈希碰撞**
4. **卡了 6-7 分钟**，一度认为面试要完蛋了（"15分钟写完我想个思路想个 6、7 分钟不完蛋了"）
5. **关键突破**：在面试官的引导下发现...

这是在面试官引导下发现的方法：
- 面试官问："字母异位词有什么共同特点？"
- 我答："它们包含的字母完全一样，只是顺序不同"
- 面试官问："如果我想让 'eat'、'tea'、'ate' 变成同一个字符串，应该怎么处理？"
- 我恍然大悟："排序！"
```

❌ **不好的示例**（太干瘪）：
```markdown
核心思想：用哈希表优化暴力法。

算法流程：
1. 遍历字符串
2. 对每个字符串排序
3. 用哈希表记录
```

#### 5.3 提取的关键信息点

从对话历史中提取：

1. **用户的原话**（直接引用）
   - "最自然的想法是..."
   - "我简单的分析了一下..."
   - "但是可惜，我没找到这样的hash 函数"
   - "然后就过了 6，7 分钟了。要是面试，15分钟写完我想个思路想个 6、7 分钟不完蛋了"

2. **面试官的引导**（如果有）
   - 关键问题："字母异位词有什么共同特点？"
   - 渐进式提示的层级

3. **用户的情绪和焦虑**
   - 卡住时间的描述
   - "完蛋了"的心情
   - "恍然大悟"的时刻

4. **代码的演进**
   - 初次尝试的代码（可能有 bug）
   - 编译错误和修正
   - 最终通过的代码

#### 5.4 交互式迭代生成内容

**不要一次性填充所有内容**，遵循以下原则：

1. **先搭建骨架**：创建文件，填入题目描述，创建思路框架
2. **根据对话历史填充**：回溯当前会话的历史对话，按时间顺序提取所有思路
3. **情景化表达**：用第一人称（"我"），记录真实思考过程
4. **标注失败尝试**：明确标记"❌ 失败尝试"，解释为什么失败
5. **记录关键对话**：面试官的引导、用户的顿悟时刻

**交互式迭代示例**：
```
AI: 已创建题解文件，基于历史对话提取了 4 个思路：
    ✅ 思路一：暴力法 - "最自然的想法"（已填充）
    ❌ 思路二：自定义哈希函数 - 失败尝试，卡了 6-7 分钟（已填充）
    ✅ 思路三：哈希表 + 排序 - 面试官引导下的突破（已填充）
    ✅ 思路四：哈希表 + 计数签名 - 进一步优化（已填充）

    题解已按模板格式生成，记录了完整的思考历程和情景细节。

    查看文件：/home/yuming/.leetcode/docs/49. 字母异位词分组/题解.md
```

## 代码注释规范

- 使用中文注释
- 关键步骤必须添加行内注释
- 复杂逻辑需要单独注释块说明

示例：
```cpp
// 使用排序后的字符串作为 key，将字母异位词分组
// 相同字母异位词排序后结果相同
unordered_map<string, vector<string>> groups;

for (string& s : strs) {
    string key = s;
    sort(key.begin(), key.end());  // 排序作为 key
    groups[key].push_back(s);      // 分组
}
```
